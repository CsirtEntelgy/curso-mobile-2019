**¿Qué y por qué OWASP?**

![](images/m11/OWASP_logo.png "OWASP-mobile-security-projects")
[OWASP](https://www.owasp.org/) (acrónimo de Open Web Application Security Project, en inglés ‘Proyecto abierto de seguridad de aplicaciones web’)
**Es un proyecto de código abierto dedicado a determinar y combatir las causas que hacen que el software sea inseguro**
La Fundación OWASP es un organismo sin ánimo de lucro que apoya y gestiona los proyectos e infraestructura de OWASP.
**Existen diversos projectos de OWASP**, destinados a la seguridad de las aplicaciones móviles que veremos a continuación.

Top Ten Mobile

![](images/m11/OWASP-Mobile-Top-10.png)
[El Proyecto de seguridad móvil OWASP Top Ten](https://www.owasp.org/index.php/Mobile_Top_10_2016-Top_10) es un recurso centralizado.
**Destinado a brindar a los desarrolladores y equipos de seguridad los recursos que necesitan para crear y mantener aplicaciones móviles seguras.**
A través del proyecto, **nuestro objetivo es**: * Clasificar los riesgos de seguridad móvil. * Proporcionar controles de desarrollo para **reducir su impacto o probabilidad de explotación**
Clasificación del **M1 al M10 las vulnerabilidades móviles que tienen mayor incidencia**.

**Resumen OWASP Mobile Top Ten**

1. Indica las vulnerabilidades de mayor incidencia. 2. Se enfoca en "areas de riesgo" y no en vulnerabilidades. 3. Adoptar este documento genera conciencia dentro de la organización. 4. Garantiza que sus aplicaciones minimicen los riesgos
Adoptar el OWASP Top 10 es quizás el primer paso, más efectivo para cambiar la cultura de desarrollo de software dentro de la organización, para poder desarrollar código más seguro.

Mobile Application Security Verification Standard (MASVS)

![MASVS](images/m11/MASVS/masvs-mini-cover.jpg)
El **[MASVS](https://github.com/OWASP/owasp-masvs/releases/download/1.1.3/OWASP_Mobile_AppSec_Verification_Standard_1.1.3_Document-es.pdf)** **es un estandar que define dos niveles estrictos de verificación de seguridad (L1 y L2)**, así como también un conjunto de requisitos de resistencia a la ingeniería inversa **(MASVS-R)**
**Se enumera de V1 a V8 diversas metricas para L1,L2 y R según corresponda a la aplicación**.
Los niveles **MASVS-L1** y **MASVS-L2** contienen **requerimientos genéricos de seguridad para todas las aplicaciones móviles (L1)** y para aplicaciones que **manejan datos altamente sensibles (L2).**
**MASVS-R cubre los controles de seguridad adicionales que se pueden aplicar**, si la prevención de las amenazas del lado del cliente son un objetivo de diseño.
![OWASP-MASVS](images/m11/MASVS/masvs-levels-new.jpg "OWASP-MASVS")

A un mayor nivel de detalle sería

#### MASVS-L1: Seguridad Estándar * Una aplicación móvil que logra el nivel **MASVS-L1** se adhiere a las mejores prácticas de seguridad en aplicaciones móviles. * Cumple con los requerimientos básicos en términos de calidad de código, manejo de los datos sensibles e interacción con el entorno móvil.
#### MASVS-L2: Defensa en Profundidad * **MASVS-L2** introduce controles de seguridad avanzados que van más allá de los requisitos estándar. * Para cumplir con el nivel L2, debe existir un modelo de amenaza y la seguridad debe ser una parte fundamental de la arquitectura y el diseño de la aplicación.
* Este nivel es apropiado para aplicaciones que manejan datos sensibles e información sensible de usuarios.
#### MASVS-R: Resistencia contra la ingeniería inversa y la manipulación * La aplicación que cuenta con el nivel de seguridad específico (L1/L2) y también es resistente a ataques específicos ( R ).
* Controles definidos del lado del cliente, e incluyen la alteración, modificación o ingeniería inversa para extraer código o datos sensibles. * Esta aplicación aprovecha las características de seguridad del hardware o bien técnicas de protección de software suficientemente fuertes y verificables.
Dentro del documento de **[MASVS](https://github.com/OWASP/owasp-masvs/releases/download/1.1.3/OWASP_Mobile_AppSec_Verification_Standard_1.1.3_Document-es.pdf)** se pueden ver los requerimientos, que van desde el V1 al V8 cubriendo todas las necesidades anteriormente descriptas.
* *V1: Requerimientos de Arquitectura, Diseño y Modelado de Amenazas* * *V2: Requerimientos en el Almacenamiento de datos y la Privacidad* * *V3: Requerimientos de Criptografía* * *V4: Requerimientos de Autenticación y Manejo de Sesiones*
* *V5: Requerimientos de Comunicación a través de la red* * *V6: Requerimientos de Interacción con la Plataforma* * *V7: Requerimientos de Calidad de Código y Configuración del Compilador* * *V8: Requerimientos de Resistencia ante la Ingeniería Inversa*

Un ejemplo de la guía sería:

**[V2: Requerimientos en el Almacenamiento de datos y la Privacidad](https://github.com/OWASP/owasp-masvs/blob/master/Document-es/0x07-V2-Data_Storage_and_Privacy_requirements.md)**

Resumen MASVS

* **Usar como una métrica** - Para proporcionar un *estándar de seguridad* el cual, las aplicaciones móviles existentes pueden ser comparadas por desarrolladores y los propietarios de las aplicaciones;
* **Utilizar como guía** - Proporcionar una guía durante todas las fases del desarrollo y prueba de las aplicaciones móviles;
* **Usar durante una auditoría de seguridad** - Proporcionar una línea de base para la verificación de seguridad de aplicaciones móviles.
* **Complemento con el checklist** - Busca evidenciar de forma metodológica y clara los puntos observados.

MSTG - Mobile Security Testing Guide - ![MSTG](images/m11/MSTG/mstg-cover-release-small.jpg)
**El [MSTG](https://github.com/OWASP/owasp-mstg) es un manual completo para pruebas de seguridad de aplicaciones móviles e ingeniería inversa.**
Describe procesos técnicos, para verificar los controles enumerados en [(MASVS)](https://github.com/OWASP/owasp-masvs/blob/master/Document-es/0x03-Using_the_MASVS.md).
Abarca conceptos generales y dependiendo de los mismos, son detallados de manera exhaustiva en cada sección.
Esta orientado para **iOS** y **Android**.
